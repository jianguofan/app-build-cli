# 架构设计决策：自建打包平台 vs Jenkins

## 1. 概述

本文档记录 App Build Platform 架构设计中的关键决策，重点说明为什么选择自建而非 Jenkins，以及其优势、劣势和未来规划。

---

## 2. 系统定位

App Build Platform 是一个 **移动端 App 构建与发布平台**，当前阶段的核心功能：

| 功能 | 说明 |
|---|---|
| 构建 | 在 macOS 上执行 Flutter 构建脚本，产出 IPA/APK |
| 发布 | 自动上传到 App Store Connect、国内各安卓应用商店、蒲公英 |
| 凭证管理 | Web 界面管理各应用商店的 API 密钥和凭证 |
| 动态配置 | 管理员可自定义构建参数选项，前端表单动态渲染 |

**目标用户**：小规模开发团队（5-15 人），需要快速打包和发布到国内多渠道。

**远期愿景**：演变为 **AI 驱动的智能发布平台**——代码提交后自动触发代码审查、单元测试、变更影响分析、智能风险评分，达标的变更自动进入构建-发布管线，不达标的打回并附带 AI 修改建议。整个流程在一个平台上闭环，不需要开发者手动穿梭于 GitHub / Jenkins / 测试平台之间。

---

## 3. 为什么不用 Jenkins

### 3.1 Jenkins 的方案长什么样

如果用 Jenkins 实现同样的功能，架构如下：

```
用户触发构建
    ↓
Jenkins Job（参数化构建）
    ├── Groovy Pipeline
    │   ├── Stage: 拉代码 + checkout 分支
    │   ├── Stage: 执行 build_app.sh
    │   ├── Stage: 收集产物（Archive Artifacts）
    │   └── Stage: 调用 fastlane 发布
    ├── Credentials Plugin（管理 API 密钥）
    ├── Build History（构建历史）
    └── 前端（Jenkins 原生 UI 或自建）
```

### 3.2 Jenkins 能做的事

| 能力 | Jenkins 实现 | 复杂度 |
|---|---|---|
| 构建触发 | Job / Pipeline，手动或 Webhook | 低 |
| 构建参数 | 参数化构建 + Active Choices Plugin | 中 |
| 执行脚本 | Pipeline `sh` step | 低 |
| 产物归档 | Archive Artifacts（内置） | 低 |
| 任务队列 | Agent Executor（内置） | 低 |
| 凭证管理 | Credentials Plugin（加密存储） | 中 |
| 构建历史 | Build History（内置） | 低 |
| 通知 | Email / Slack Plugin | 低 |
| 分布式 | Master + Agent（内置） | 中 |

### 3.3 Jenkins 的痛点

**1. 用户体验差**

Jenkins 原生 UI 面向运维人员，而非内部用户（产品经理、测试同事）。要实现 Ant Design 级别的交互（动态表单、实时日志、发布状态卡片），需要：
- 开发 Jenkins Plugin（Java/Groovy，API 极其难用），或
- 在 Jenkins 外面再套一层前端，增加维护复杂度

**2. 动态选项能力弱**

本系统允许 Admin 在 Settings 页面动态增删构建参数（如新增 flavor、feature flag），前端表单自动渲染。Jenkins 要实现同样效果需要：
- Active Choices Plugin + 脚本化参数
- 参数变更需要重新运行 Job 配置
- 无法做到"保存即生效"

**3. 发布管理无原生支持**

- 构建 + 发布一体化是本系统的核心需求
- Jenkins 的 Pipeline 把构建和发布写在一起，但发布的选择（发哪些渠道）、记录追踪、重新发布这些概念 Jenkins 都没有
- 各国内应用商店的上传凭证管理（小米/华为/OPPO/VIVO等，每个平台的认证方式不同）需要大量自定义代码

**4. Groovy 开发的痛苦**

- Jenkins Pipeline 用 Groovy 编写，调试体验差
- 语法约束多（序列化限制、沙箱限制）
- 难以单元测试
- 团队技术栈是 TypeScript，维护 Groovy 代码需要额外学习成本

**5. 运维成本不低**

- Jenkins 也需要运维：Plugin 升级兼容性、JVM 调优、备份恢复
- 官方推荐至少 4GB 内存，插件多了更重
- macOS Agent 配置同样复杂

## 4. 自建方案

### 4.1 技术架构

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  React SPA   │────▶│  NestJS API  │────▶│   Redis      │
│  (Ant Design)│     │  (TypeScript)│     │  (Bull Queue)│
└──────────────┘     └──────┬───────┘     └──────────────┘
                            │
                    ┌───────▼────────┐
                    │  StorageService │
                    │  (In-Memory)    │
                    └───────┬────────┘
                            │
              ┌─────────────┼─────────────┐
              ▼             ▼             ▼
       ┌──────────┐ ┌──────────┐ ┌──────────────┐
       │ Executor │ │ Fastlane │ │ Publishing   │
       │ (spawn)  │ │ Publisher│ │ Credentials  │
       └──────────┘ └──────────┘ └──────────────┘
```

### 4.2 核心组件职责

| 组件 | 职责 |
|---|---|
| **Bull Queue** | 构建/发布任务队列，防并发冲突，支持重试 |
| **Executor** | 本地执行 `build_app.sh`（shell spawn），流式输出日志 |
| **FastlanePublisher** | 封装各应用商店上传逻辑，通过 `bundle exec fastlane` 执行 |
| **PgyerPublisher** | 直接 HTTP multipart POST 上传到蒲公英 |
| **StorageService** | Phase 1 内存存储，Phase 2 迁移到数据库 |
| **WebSocket Gateway** | 实时推送构建日志和状态更新到前端 |

### 4.3 核心数据流

```
用户提交构建 → BuildService 创建 BuildTask → Bull Queue
    → BuildProcessor.handleBuild
        → Executor.localExecute(build_app.sh)
        → 实时日志推送到前端
        → 构建完成 → PublishService.publish
            → 为每个目标渠道创建 PublishRecord
            → Bull Queue → PublishProcessor.handlePublish
                → Fastlane upload / Pgyer upload
```

## 5. 自建优劣势对比

### 5.1 优势

| 维度 | 说明 |
|---|---|
| **用户体验** | React + Ant Design 定制 UI，完全按需求设计：NewBuild 动态表单、实时日志流、发布状态卡片、凭证管理 Modal |
| **迭代效率** | 全栈 TypeScript，前后端同语言。加一个发布渠道只需改 Fastfile + 后端配置，半小时上线 |
| **轻量部署** | 4 个 Docker 容器（Nginx + Frontend + Backend + Redis），docker-compose up 即可 |
| **完全可控** | 构建环境是宿主机 macOS（Xcode 编译 iOS 必需），不存在 Jenkins Agent 配置兼容问题 |
| **领域专注** | 专为"Flutter 打包 + 发到国内应用商店"设计，无冗余功能 |
| **数据模型清晰** | BuildTask / PublishRecord / PublishingCredential 直接映射业务实体，无 Jenkins 的 Job / Run / Node 抽象层 |

### 5.2 劣势

| 维度 | 说明 |
|---|---|
| **轮子重复** | 队列、任务状态管理、历史记录等 Jenkins 内置功能需自行实现 |
| **数据可靠性** | Phase 1 内存存储（`Map`），重启丢失构建历史 |
| **分布式能力** | 仅单机本地执行，缺乏 Agent 管理 |
| **权限系统** | 仅硬编码单用户，无 RBAC |
| **生态缺失** | 无 Jenkins 插件生态（通知、监控、备份、集成等） |
| **运维能力** | 无内置 Prometheus 指标、Build Failure 分析、SLA 统计 |

## 6. 硬核论据：为什么自建是天选方案

前面的章节分析了优劣势，但那些偏"理论"。这一章给出几个硬核论据——如果你还在纠结，看完这些再做决定。

### 6.1 论据一：Jenkins 不解决你的硬问题

很多人选 Jenkins 是因为"Jenkins 能把构建跑起来"。问题是你真正的**硬问题**根本不在 Jenkins 的能力范围内：

```
你的硬问题（詹金斯不解决）：                        詹金斯能解决的问题：
┌─────────────────────────────┐                 ┌──────────────────────┐
│  各国内应用商店的认证和上传    │                 │  任务排队（Bull也做）  │
│  （小米/华为/OPPO/VIVO/应用宝 │                 ├──────────────────────┤
│   的API凭证格式完全不同）      │                 │  构建历史（Map也做）  │
├─────────────────────────────┤                 ├──────────────────────┤
│  Fastlane 集成与调用          │    ← 但是 →    │  基础鉴权（JWT也做）   │
├─────────────────────────────┤                 ├──────────────────────┤
│  构建参数的动态配置和管理      │                 │  日志（WebSocket也做） │
├─────────────────────────────┤                 └──────────────────────┘
│  发布记录追踪与重新发布        │                       ↑
├─────────────────────────────┤                  这些全部加起来
│  凭证管理的Web交互界面        │                  只占工作量的 ~20%
├─────────────────────────────┤
│  实时构建日志推送              │
├─────────────────────────────┤
│  每个平台独立配置状态展示      │
└─────────────────────────────┘
        ↑
  这些占工作量的 ~80%
  詹金斯一个都不帮你解决
```

**结论：选了 Jenkins，80% 的代码一样要写**——而且是用更痛苦的 Groovy 在 Jenkins 的约束下写。最终工作量和自建相差不大，但开发体验差了十倍。

### 6.2 论据二：Plugin 陷阱——你以为有插件，其实没有

Jenkins 的核心理念是"有插件生态"。你把实际需求过一遍看看：

| 需求 | 有现成 Plugin 吗？ | 实际情况 |
|---|---|---|
| **构建排队** | ✅ 有，Jenkins 自带 | 一个 Job 一个队列，够用 |
| **SSH 执行** | ✅ 有，SSH Pipeline Steps | 够用 |
| **参数化构建** | ✅ 有，built-in | 勉强能用，但不如动态表单 |
| **发布到小米商店** | ❌ **没有！** | 你要自己写（Java Plugin 或 Pipeline 里调 curl） |
| **发布到华为商店** | ❌ **没有！** | 同上 |
| **发布到 OPPO 商店** | ❌ **没有！** | 同上 |
| **发布到 VIVO 商店** | ❌ **没有！** | 同上 |
| **发布到 360 商店** | ❌ **没有！** | 同上 |
| **发布到应用宝** | ❌ **没有！** | 同上 |
| **发布到荣耀商店** | ❌ **没有！** | 同上 |
| **发布到三星商店** | ❌ **没有！** | 同上 |
| **蒲公英上传** | ❌ **没有！** | 同上 |
| **发布记录管理** | ❌ **没有！** | 同上 |
| **动态构建选项 UI** | ❌ **没有！** | 同上 |
| **凭证管理 Web UI** | ❌ **没有！** | Jenkins 只有 Key-Value 存储，没有按平台定制表单 |
| **实时日志前端** | ❌ **没有！** | Jenkins 自己的日志页面，不是你想要的 React 组件 |

**Jenkins Plugin 生态解决的是通用 CI/CD 问题（Git 集成、Slack 通知、SonarQube 等）。你的所有领域特定需求，一个插件都没有。全部自己写。**

更糟的是，一旦你开始写 Jenkins Plugin，你陷入的是：
- Java 代码（不是你的 TypeScript 技术栈）
- Plugin API 极其不友好（大量的 XML 配置、低版本的 Java 约束）
- Plugin 和 Jenkins 核心版本有兼容性要求
- 写出来的 Plugin 只在这个 Jenkins 版本能用

### 6.3 论据三：Jenkins on macOS 是公认的坑

你需要的是 macOS 上构建 iOS（因为 Xcode 编译），这是 Jenkins 社区公认的痛点：

| 问题 | Jenkins 方案 | 自建方案 |
|---|---|---|
| **Xcode 版本管理** | 需要 xcode-select + 额外配置，切换版本需要手动或写复杂脚本 | `executor.service.ts` 中直接指定编译环境 |
| **代码签名** | Keychain 在 headless SSH 会话中频繁"消失"，开发者经常被"code sign error"折磨 | 代码中 `security unlock-keychain` 自动处理 |
| **Agent 不稳定** | macOS 做 Jenkins Agent 经常因为睡眠、系统更新、网络变化掉线 | 没有 Agent 层，直接在宿主机 spawn process |
| **资源占用** | Jenkins Master + Agent 两个 JVM 进程吃掉 2G+ 内存 | 一个 Node 进程 + Bull + Redis，~200MB |
| **升级重启** | macOS Agent 上的 Jenkins 升级麻烦，Plugin 冲突可能导致服务不可用 | `git pull && docker-compose up -d` |

**Jenkins on macOS 的运维痛苦，你去搜索一下 "jenkins mac agent code signing nightmare" 就能感受到。**

### 6.4 论据四：你还嫌代码不够多？——把已写好的代码扔掉重写？

这是最务实的一点：**你已经写完了**。

回看现在的代码库：
- 后端：构建编排、队列管理、10 个平台的发布逻辑、凭证存储、WebSocket 实时推送、动态配置 API
- 前端：5 个页面 + 动态表单 + 日志查看器 + 发布状态卡片 + 凭证管理 UI
- Fastlanefile：256 行，9 个 lane，覆盖所有渠道

**这些代码全在 git 里，功能已经在跑。换成 Jenkins 意味着全部扔掉，用 Groovy/Java 重写一遍。**

这不是"选 A 还是选 B"的问题，这是"已经有了 A，要不要把它改成 B"的问题。而 B 并没有解决任何 A 解决不了的问题。

### 6.5 论据五：Jenkins 的使用者是谁？——你的团队不是 DevOps 团队

```
Jenkins 的设计对象：
┌──────────────────────────────────────────┐
│  "假设你是一个专职的 DevOps 工程师，       │
│   你的日常工作就是维护 CI/CD 管线，        │
│   你熟悉 Groovy、JVM 调优、Plugin 管理"   │
└──────────────────────────────────────────┘

你的团队画像：
┌──────────────────────────────────────────┐
│  "Flutter 客户端开发为主，需要快速打包，    │
│   没有专职 DevOps，不想学一套新的工具链，   │
│   产品的需求经常变，希望改配置像改网页一样" │
└──────────────────────────────────────────┘
```

**工具选择要看使用者的能力模型，而不是功能对比表。**

你的团队的核心能力在客户端（Flutter/iOS/Android），不是 DevOps。让客户端工程师去维护 Jenkins Pipeline 还是去维护 NestJS + React？后者的学习曲线平缓一万倍。

### 6.6 论据六：机会成本——你学到的东西带不走

选 Jenkins 的隐性成本：

| 投入 | 在 Jenkins 上学到的东西 | 在自建平台上学到的东西 |
|---|---|---|
| **时间** | Groovy Pipeline 语法 | TypeScript 全栈开发 |
| **精力** | Plugin 兼容性踩坑 | NestJS 架构设计 |
| **经验** | Jenkins 运维技巧 | Bull 队列、WebSocket、Docker |
| **价值** | 下一个公司大概率不用 Jenkins | 全栈 + 系统设计能力，到哪都用得上 |

**Jenkins 的知识是封闭的、不可迁移的。** TypeScript + React + NestJS + Docker 是行业通用技能，即使这个项目停了，这些技能也永远是你的。

### 6.7 论据七：规模不匹配——用货车运一瓶水

Jenkins 官方推荐的配置：
- 4GB+ 内存
- 20GB+ 磁盘
- 如果是 Master + Agent 架构，还要更多

你的实际需求：
- 一台 macOS 机器
- 几个人用
- 同时只有一个构建在跑

```
Jenkins 架构：
┌─────────┐     ┌──────────┐     ┌──────────┐     ┌──────────────┐
│ Jenkins │     │ Jenkins  │     │ Database │     │  各种Plugin  │
│ Master  │────▶│ Agent    │────▶│ (Derby)  │────▶│   (30+个)    │
│ (JVM)   │     │ (JVM)    │     │ (XML)    │     │  (JVM堆)      │
└─────────┘     └──────────┘     └──────────┘     └──────────────┘
                  总内存占用：~3-5GB

你的架构：
┌──────────┐     ┌──────────┐
│  Node.js │     │  Redis   │
│  (~80MB) │────▶│  (~5MB)  │
└──────────┘     └──────────┘
  总内存占用：~200MB
```

**Jenkins 是为大型组织的复杂 CI/CD 网络设计的，你的场景用不到它 80% 的能力，却要承受它 100% 的重量。**

### 6.8 论据八：Jenkins 的生态在衰落

数据不会说谎：
- GitHub Actions 2019 年发布后，CI/CD 市场快速从 Jenkins 迁移
- GitLab CI 内置、GitHub Actions 内置——"不需要额外运维一个 CI 服务器"成为主流
- Jenkins 的 GitHub Star 增长已经停滞多年，Issues 数量却在增加
- 活跃 Plugin 开发者年年在减少
- Jenkins 的官方推荐基准在新项目中已逐渐转向"维护模式"

**2025/2026 年从头开始一个 Jenkins 项目，就像 2015 年从头开始一个 SVN 项目：能用，但方向是反的。**

## 7. 为什么这比 Jenkins 更关键：AI 驱动管线

这是整个决策中最重要的一环。如果远期目标是**在发布前自动完成代码审查、单元测试、变更影响评估、风险评分**，那么 Jenkins 和 GitHub 的能力边界就非常明显了。

### 7.1 未来管线长什么样

```
开发者提交 PR
    ↓
① AI Code Review（Claude/GPT 自动审查 diff）
    ├── 发现代码问题 + 给出修改建议
    ├── 严重问题 → 打回，开发者按建议修改
    └── 通过 → 进入下一阶段
    ↓
② 智能单元测试
    ├── AI 分析变更影响范围，选择需要运行的测试用例
    ├── 运行测试 + 收集覆盖率
    └── 失败 → AI 分析失败根因，给出修复建议
    ↓
③ 变更风险评估
    ├── AI 分析：改动范围、历史 Bug 模式、测试覆盖率缺口
    ├── 综合评分（低/中/高/极危）
    └── 高风险 → 标记需要人工审核 + 附加测试
    ↓
④ 构建 + 发布决策
    ├── 低风险：自动构建 → 发布到指定渠道
    ├── 中风险：自动构建 → 人工确认后发布
    └── 高风险：打回 + AI 修改建议报告
    ↓
⑤ 反馈闭环
    ├── AI 记录每次决策结果
    ├── 学习历史发布质量数据
    └── 持续优化风控模型
```

### 7.2 Jenkins 能做这些吗？

| AI 场景 | Jenkins 能力 | 能否实现 |
|---|---|---|
| **AI Code Review** | Pipeline 中加 `sh 'curl https://api.xxx/review'` 调用 AI API，解析结果判断是否阻断 | ⚠️ 勉强能做，但：无 review UI、无法展示差异对比、无法交互式确认修改建议 |
| **智能测试选择** | 同样可以写脚本分析 diff → 选测试 → 调用 AI API | ⚠️ 能做，但测试选型逻辑维护在 Groovy 中，无法可视化查看覆盖率缺口 |
| **风险评分和决策** | 可以写脚本计算评分 → 根据阈值 fail pipeline | ⚠️ 能做，但评分逻辑是黑盒，无法给开发者提供可视化的风险报告 |
| **反馈闭环 / AI 模型训练** | Jenkins 没有数据管道，日志是纯文本，难以结构化存储和回放 | ❌ 不适合 |
| **交互式 UI（审查面板、风险仪表盘）** | Jenkins 原生 UI 不行，需要额外自建前端 | ❌ 不适合 |

**结论：Jenkins 本质上是一个任务执行引擎，不是智能平台。** 你可以在 Pipeline 里加脚本调 AI API，但：
- 所有 AI 能力都要靠 Groovy 胶水代码拼凑
- 没有交互式 UI（不能展示 diff review、不能可视化测试覆盖率缺口、不能拖动调整风险阈值）
- 数据是割裂的（构建日志、测试结果、AI 评分散落在不同地方）
- 反馈闭环几乎不可能（Jenkins 不是数据平台）

### 7.3 GitHub Actions 能做这些吗？

GitHub Actions 比 Jenkins 现代一些，但同样受限于 YAML 工作流的表达能力：

| 能力 | 说明 |
|---|---|
| **调用 AI API** | YAML step 中 `curl` 或 GitHub Actions `run:`，可以调，但结果处理笨重 |
| **代码审查集成** | GitHub 本身有 Code Review UI，但 AI 审查结果需要以 comment 形式注入，交互较弱 |
| **测试矩阵** | GitHub 的 matrix 策略跑测试很方便 |
| **状态检查** | 可以将 AI 评分作为 required check，阻断合并 |
| **局限** | YAML 不是编程语言；复杂条件逻辑难以维护；无法做持久化数据分析和模型训练 |

**关键问题：GitHub 是一个代码托管平台，不是一个发布平台。** 
- 它没有"发布到小米应用商店"的概念
- 它没有"选择发哪些渠道"的能力
- 它没有"发布记录 + 审核状态追踪"的数据模型
- 它不能管理国内应用商店的 API 凭证

### 7.4 自建平台为什么是更好的 AI 载体

```
Jenkins / GitHub Actions:                   自建平台:
┌────────────────────────┐                 ┌────────────────────────┐
│  执行引擎（Execute）     │                 │  智能平台（Intelligence）│
│                        │                 │                        │
│  Groovy / YAML 描述流程 │                 │  TypeScript 定义全流程   │
│  调 API 勉强能做        │                 │  原生集成 AI SDK        │
│  数据是副产品           │                 │  数据是一等公民          │
│  没有持久化 AI 状态      │                 │  可建 AI 反馈闭环       │
│  UI 不可控              │                 │  前端完全可控            │
└────────────────────────┘                 └────────────────────────┘
```

自建平台在 AI 场景下的核心优势：

**1. 数据完整性**

Jenkins 做 AI 最大的问题是：数据是零散的。构建日志在 Jenkins、测试结果在另一个平台、代码 Review 在 GitHub、发布状态在 App Store Connect。AI 模型需要完整的数据链来做分析和预测，数据不在一个地方，AI 的效果就是割裂的。

自建平台的 BuildTask → CodeReview → TestRun → PublishRecord 是一条完整的数据链，AI 可以在这条链上端到端地学习。

**2. 交互体验可控**

AI Code Review 不只是"返回一段评语"——它需要：
- diff 可视化 + 问题标注 + 修改建议
- 开发者可以点赞/点踩（反馈信号）
- 管理者可以看团队级别的 AI 审查报告
- 这些在 Jenkins 里做不了

**3. 反馈闭环**

这是 AI 持续改进的核心：
```
AI 给出审查意见 → 开发者接受或拒绝 → 记录结果
    → 分析哪些建议被采纳、哪些被忽略
    → 优化 prompt / 微调模型
    → 下一轮更准确
```
自建平台可以构建这个闭环，Jenkins 不能。

**4. 安全与隐私**

代码发给外部 AI API 做审查是敏感操作。自建平台可以：
- 控制哪些文件发给 AI（排除密钥文件、配置文件）
- 实现本地模型部署（如 Ollama + CodeLlama）
- 审计所有 AI 调用记录
- 这些在 Jenkins Pipeline 里难以精细控制

## 8. 决策依据

### 8.1 关键判断

```
问题：需要分布式构建吗？
答案：不需要——只有一台 macOS 打包机

问题：需要多用户权限管理吗？
答案：暂时不需要——小团队内部使用

问题：需要和 Jira / GitHub / Slack 深度集成吗？
答案：可以后续加——不构成 Jenkins 的刚需

问题：构建参数频繁变化吗？
答案：是的——经常需要加 feature flag 和构建参数

问题：业务逻辑独特吗？
答案：非常独特——国内应用商店发布流程是强领域逻辑

问题：未来要集成 AI Code Review / 智能测试 / 风险评分吗？
答案：是的——这是决定性的长线规划
```

### 8.2 结论

**自建是当前阶段的正确选择，理由：**

1. **场景匹配**：Jenkins 解决的是通用 CI/CD 问题，而本系统只需要"构建 + 发布"这一条垂直场景。为一个毛坯房装四层别墅的脚手架，不划算。

2. **最大痛点 Jenkins 不解决**：发布到国内应用商店的凭证管理、Fastlane 调用、发布记录追踪，Jenkins 插件生态不做，都得自己写。

3. **AI 集成是决定性的长线理由**：如果未来目标是 AI 驱动的智能发布管线——自动 Code Review、智能测试选择、变更风险评估、反馈闭环——那么 Jenkins 和 GitHub Actions 都不是正确的平台。它们是"执行引擎"，不是"智能平台"。自建平台完全拥有数据模型、前端 UI、SDK 集成能力，才能支撑 AI 管线的演进。

4. **迭代速度优先**：NestJS + React 的迭代速度远快于 Jenkins Pipeline + Groovy + Plugin 开发。

5. **运维成本相当**：Jenkins 本身也需要 JVM 调优、Plugin 兼容管理，并不比 4 个 Docker 容器低。

## 9. 自建平台对团队的独特价值

这一章从**团队成长**和**技术资产积累**的角度分析自建的深层价值——不仅仅是"能不能做"，而是"做了之后团队能获得什么"。

### 9.1 技术栈统一，降低团队认知负担

| 维度 | Jenkins 方案 | 自建方案 |
|---|---|---|
| **后端语言** | Groovy（Pipeline）+ Java（Plugin） | TypeScript（NestJS） |
| **前端** | 无 / 被迫自建 | React + TypeScript |
| **构建定义** | Jenkinsfile / Groovy DSL | 后端代码（普通 TS 函数） |
| **数据存储** | Jenkins 内置 DB（XML 文件） | SQL / 对象存储 |
| **学习曲线** | 需要 Jenkins 专项经验 | 普通全栈工程师即可上手 |

**实际影响**：团队不需要一个"懂 Jenkins 的人"，任何一个带着 TypeScript 技能的工程师都能贡献。招聘成本降低、沟通成本降低、Code Review 成本降低。

### 9.2 快速迭代，业务响应快

自建平台的一个典型迭代场景：

```
产品：下周要加一个「staging2」环境

Jenkins 方案：
1. 找到 Jenkins Job 配置页面
2. 修改参数化构建的 Choice 列表
3. 重新运行 job 才能看到效果
4. 如果涉及前后端 UI，还要改前端代码
5. 测试 → 可能需要重启 Jenkins
→ 耗时：几小时到几天

自建方案：
1. 管理员在 Settings 页面的「环境」选项里新增「staging2」
2. 保存即生效
→ 耗时：30 秒
```

另一个场景：

```
产品：帮我把 login 模块打个 debug 包发到蒲公英临时测试

自建方案：
1. 打开平台 → 新建构建 → 参数选好 → 提交
2. 构建完 PDF 飞书链接发出去
→ 耗时：1 分钟

Jenkins 方案：
1. 打开 Jenkins → 找到对应 Job
2. 填参数（如果参数化了）
3. 构建 → 去工作空间找 APK
4. 手动上传蒲公英 → 复制链接
→ 耗时：5-10 分钟，且容易出错
```

### 9.3 代码即配置，版本可控

Jenkins 的问题：
- Job 配置存在 Jenkins 的 XML 文件里，不同步到 Git
- 谁改了什么、什么时候改的，没有审计
- 恢复环境需要手动重建 Job

自建方案：
- 所有业务逻辑是 TypeScript 代码，Git 管理
- 前端 UI、后端 API、构建流程、发布逻辑都在同一个仓库
- 一个 `git log` 就能看到整个平台的演进历史
- `git checkout` 一个 tag，就能部署到任何历史版本

### 9.4 团队的技术资产积累

Jenkins 是黑盒，你在上面做的配置、Pipeline 脚本离开 Jenkins 就没有价值。

自建平台每一行代码都是团队的技术资产：

| 资产类型 | 价值 |
|---|---|
| **NestJS 后端代码** | 可复用：构建编排、队列管理、WebSocket 实时推送等模式可以用于其他内部工具 |
| **React 前端组件** | 可直接复用到其他管理后台：动态表单、日志查看器、实时状态卡片 |
| **Fastlane 集成经验** | 各国内应用商店的上传 API 对接经验是行业 Know-how |
| **AI 集成管道** | Code Review + 风险评分的架构设计可以扩展到其他 AI 场景 |
| **自动化测试框架** | 未来集成单元测试时，测试框架的选型和配置经验沉淀在项目里 |

**每个功能迭代都是在为团队累积能力，而不是给 Jenkins 的墓碑添砖。**

### 9.5 团队自主权与决策空间

自建平台意味着：

- **不依赖第三方**：Jenkins 升级、Plugin 弃用、安全漏洞，这些都不用操心
- **功能优先级自主**：想加什么功能就加什么，不在 Roadmap 上等 Jenkins 社区
- **问题可修**：遇到 bug，Git 拉下来修，不用等 vendor 发 hotfix
- **深度定制**：可以做任何团队想要的交互——不是"Jenkins 能不能做"，而是"我们想不想做"

### 9.6 AI 集成带来的团队成长红利

未来引入 AI 能力后，团队额外获得：

| 能力 | 团队收益 |
|---|---|
| **LLM 集成经验** | 如何调 prompt、如何解析 AI 输出、如何做安全过滤——这是通用技能 |
| **数据管道建设** | 构建 → 测试 → 发布全链路数据，是团队做任何 AI 产品的数据基础 |
| **反馈闭环设计** | AI 建议 → 人类确认 → 模型优化 → 更准的 AI 建议，这个模式可以复用到所有内部工具 |
| **风险量化能力** | 从"凭感觉判断发布风险"到"数据+AI 辅助决策"，团队的数据文化会提升 |

> **总结：自建平台不只是"做一个工具"，而是在打造团队的技术能力底座。**

## 10. 未来功能规划

### Phase 2（近期可做）

| 功能 | 实现思路 | 说明 |
|---|---|---|
| **数据库持久化** | StorageService 改为 SQLite/PostgreSQL | 最紧迫，解决重启丢数据 |
| **Git Webhook** | 接收 GitHub/GitLab webhook → 自动建任务 | 减少手动操作 |
| **定时构建** | Bull Queue repeatable job | 支持 nightly 构建 |
| **多用户 + 权限** | 用户表 + JWT 角色校验 | 支持多人协作 |
| **构建通知** | 企业微信/飞书/Lark 消息推送 | 替代 Jenkins 通知 |

### Phase 3（AI 能力起步）

| 功能 | 实现思路 | AI 程度 |
|---|---|---|
| **AI Code Review** | 提交代码时自动调 Claude/GPT API 审查 diff，结果展示在平台内 | 🤖 API 调用 |
| **自动单元测试** | 分析变更 → 选择测试用例 → 运行 → AI 分析失败根因 | 🤖 API 调用 |
| **智能风险评分** | 结合代码变更量、影响范围、历史 Bug、测试覆盖给出风险分 | 🤖 API + 规则引擎 |
| **智能发布建议** | 根据风险评分自动建议发布渠道（全量/灰度/暂缓） | 🤖 API + 规则引擎 |
| **测试覆盖率看板** | 可视化展示覆盖率趋势、缺口分析 | 📊 数据可视化 |

### Phase 4（AI 深度集成）

| 功能 | 实现思路 | AI 程度 |
|---|---|---|
| **AI 修改建议自动应用** | AI 发现代码问题 → 生成 patch → 开发者一键确认 | 🧠 代码生成 |
| **智能测试生成** | AI 为新代码自动生成单元测试用例 | 🧠 代码生成 |
| **历史数据反馈闭环** | 记录每一次 AI 建议 + 人类决策 → 优化 prompt/模型 | 🧠 模型调优 |
| **变更语义分析** | AI 理解变更的"意图"而非字面 diff，发现隐蔽回归风险 | 🧠 深度分析 |
| **发布后监控关联** | 发布后的崩溃数据与变更关联，AI 分析根因 | 🧠 数据挖掘 |
| **AI 对话式构建** | "帮我把 login 模块打个 debug 包发到蒲公英"→ 自动完成 | 🧠 智能交互 |

### Phase 5（演进方向）

当系统发展到需要以下能力时，可考虑引入 Jenkins 作为执行层（上层仍用本平台编排）：

| 场景 | 方案 |
|---|---|
| 分布式 Agent（多台编译机） | 本平台调度 → Jenkins Agent 执行 |
| 多租户（多个团队共享） | 本平台做租户管理 + 权限 |
| 第三方集成（Jira/Slack/Jenkins） | 本平台做 API 层 + Webhook |
| 规模化运维（2000+ 构建/月） | 引入专用 CI/CD 执行层 |

关键原则：**AI 智能层永远在本平台，Jenkins/GitHub 只作为可选的执行后端切换。**
